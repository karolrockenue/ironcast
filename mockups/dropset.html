How it works (the proposal)

  • Any working set can become a drop set. Tap ⇊ under the current set to chain reduced-weight drops onto it. Default new drop = previous stage × 0.75 rounded to the increment (same rounding the deadlift technique-day uses).
  • It is still one set. The parent row (the top stage) is the real set_number row — it alone feeds the progression engine and PR detection. Nothing in decideProgression() / PR logic changes. Drops are extra volume hung off it.
  • No rest between drops. The rest timer is suppressed while a drop ladder is open and only fires once the final drop is logged — matching how a drop set is actually performed.
  • Volume is summed across all stages. History, Summary, and the CSV export count parent + every drop. A drop badge + the 12.5 → 10 → 7.5 ladder render inline.

Data model — additive, non-destructive

  • New table set_drops(id, set_id → sets.id, drop_seq, weight, reps). One row per drop stage below the parent.
  • Migration is additive (CREATE TABLE IF NOT EXISTS + insert the new exercise if absent) — not the current nuke-and-reseed, so your real logged history on the phone survives.
  • CSV export gains a drop_seq column (0 = top stage, 1..n = drops) so an LLM/spreadsheet can read the full ladder.
